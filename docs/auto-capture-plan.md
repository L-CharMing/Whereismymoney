# 自动记账接入方案

## 结论

### 1) 能不能直接接微信 / 支付宝 API？

现阶段更稳妥的判断是：**不能把“官方开放平台 API”作为个人记账 App 自动获取个人消费账单的主方案**。

原因：

- 支付宝开放平台当前公开入口仍然强调的是“小程序 / 第三方应用 / 网页/移动应用开发”等面向商家、服务商的开放能力，而不是个人用户账单读取能力。
- 我没有找到可靠的、面向普通个人记账 App 的“读取微信支付或支付宝个人账单”的官方开放接口说明。
- 因此这里采用的产品策略是：**把官方 API 视为潜在补充方案，而不是主链路**。

### 2) 自动记账主链路怎么做？

优先级建议：

1. **无障碍服务**：监听支付成功页或账单详情页，抓取界面文字，解析金额 / 商户 / 时间。
2. **本地规则与去重**：在本地判断是否记账，避免重复入账。
3. **可选大模型分类**：只上传必要字段做分类。
4. **无线 ADB 辅助**：用于开发、适配和高级用户模式，不作为默认长期后台方案。

## 为什么无线 ADB 不该做默认主链路

- 无线调试需要用户先在开发者选项中手动开启并配对。
- 连接状态可能受局域网、设备重启、系统策略影响，不稳定。
- 更适合：开发调试、页面结构分析、适配支付 App 页面。
- 不适合：普通用户长期日常依赖的自动记账后台。

## 当前代码实现了什么

- `PaymentAccessibilityService`：递归读取当前窗口文本。
- `PaymentCaptureManager`：对微信 / 支付宝包名和支付金额做基础解析。
- `CaptureScreen`：展示无障碍 / 无线 ADB 的产品路线和本地导入流程。

## 下一步编码建议

- 在 AccessibilityService 中将识别到的 candidate 发送给持久化层
- 加入支付成功关键字、金额提取、商户名优先级规则
- 加入 30~90 秒时间窗去重
- 为微信、支付宝分别维护页面解析模板
